Reflected Cross-Site Scripting (XSS) — Responsible Writeup

Sanitized case study • Responsible disclosure • Practical remediation
← Back to Blog
Author: Aryan Pareek • Date: Nov 9, 2025 • Tags: XSS Web Security

Summary: During authorized testing of an e-commerce web application, a reflected Cross-Site Scripting (XSS) vulnerability was identified in the product search endpoint. The input from the search parameter was being directly rendered in the HTML response, allowing execution of user-supplied scripts in the browser. The report below is fully sanitized and safe to publish.

Vulnerability Context

The affected endpoint was /search-products.php which accepted the parameter search. Upon entering crafted input, the unencoded value appeared directly in the returned page HTML. The vulnerability was safely verified under permission and responsibly reported to the concerned team.

Observed Behavior (Sanitized)

The following screenshots demonstrate the reflection behavior observed in the page. Sensitive elements and payloads have been sanitized for public viewing.

Impact

An attacker could craft a malicious link containing a script payload that, when visited by another user, would execute arbitrary JavaScript in that user’s browser. This could lead to cookie theft, user redirection, or execution of unauthorized actions within the web application.

Safe Reproduction (for Developers)

  1. Use a non-production environment or dummy test account.
  2. Check how the search query input is echoed back in the response.
  3. Implement encoding/escaping on the reflected variable and verify the response renders safely.

Root Cause

User-controlled input was rendered inside HTML without proper output encoding or sanitization. The lack of context-specific escaping (HTML/attribute) enabled script injection upon rendering.

Remediation

  • Escape & Encode: Apply HTML encoding for all user input reflected in pages.
  • Implement CSP: Use Content-Security-Policy: default-src 'self'; script-src 'self'; object-src 'none'; to block inline execution.
  • Validate Input: Restrict search parameters to safe characters and lengths.
  • Security Testing: Include automated and manual XSS detection during QA.
Note: This blog avoids including any live payloads or exploit strings. The intention is educational — showing the reflection pattern and remediation guidance for web developers.
